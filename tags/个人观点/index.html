博客换主题，不再用hexo-theme-archer

原来的主题是 https://github.com/fi3ework/hexo-theme-archer

但不幸的是，维护者有自己的事要做，老的版本有老的 bug，最新的版本有新的 bug，修复时间未知，PR里另一个维护者有不同意见却长期没有获得回应。整个主题的版本不可控制，并且它推荐的安装方式是用整个 Github 项目，配置什么的都在里面，已经开始影响我写博客了。

因为提交的时候是主题和文章一起提交。而原项目的所有配置都写死在了项目里，如果我需要使用，最好的方式是我自己去维护一个个人版本。这有几个问题。

  • 这个项目的使用方式需要使用者自行在 themes 里做一个单独的代码版本控制（项目本身并没有明确要求，但实际使用中肯定是需要的）。如果不做，等哪天你需要更新了，你会发现不知道哪些文件是不必要的，哪些文件需要更新，哪些文件被还原了导致奇怪的问题，甚至你大概需要自己重新编译一下。这是一个正常的 Hexo Theme 不会有的要求，我的主要目的是写博客，虽然个人愿意维护一个自己的版本，甚至已经这么做过了。
  • 这个自己维护的代码应该是私密的，因为配置都在里面，我不希望别人围观配置，但个人配置这种东西，它就应该在本地。
  • 调试主题以及配置会影响我更新文章。如果调试需要花几天时间甚至更长，我没法把一个有 bug 的博客更新出去，况且我本身不希望花太多时间在这上面。
  • 整个项目的设计存在明显问题，大部分其他项目提供配置示例都是加了 .sample，这样不会有配置覆盖修改之类的问题，参考上面第二点，原始 code 应该和编译后的 release 文件分开。而在这个项目中，你需要把 .git 文件夹还有一大堆应该用不上的文件都放到你的 themes 文件夹中，另外我估摸编译的 node_modules 和主题中需要使用的 node_modules 混在了一起。
  • 主要维护者删 package-lock.json

新的主题是 https://github.com/ppoffice/hexo-theme-icarus

就本身来说我还是比较喜欢原来的主题样式，新的主题还有好多我不太喜欢的变化，例如页面宽度（原来的我自己调宽了），预览的长度。不过好歹支持了 toc，虽然解析似乎还有问题

不过这块应该可以克服和改进，毕竟很多还没来得及研究

Winget

开源，MIT，开源地址 https://github.com/microsoft/winget-cli
Winget 永远的神，Win 10 开始自带
目前除了批量升级有点使用上的问题以外，其他做的都挺强

目前可以通过以下方式批量升级(也可以用来备份恢复软件列表)

  1. winget export -o 文件
  2. 删除列表里不需要的文件
  3. winget import 文件

winget 下的升级文件都在 %Temp% 里，一些软件升级时需要关闭其他软件可能会导致升级失败，可以手动运行安装包升级

请勿购买QNAP

本文属于用啥黑啥系列。
在专用NAS中，如果想用那些某类玩家都用的软件，其实可选择性不多。
像什么西数，铁威马，别人跟你说某软件可能用不了，估计你的心就凉了大半。
然后就会了解到群晖的贵。

此时留给你的选择大概不多，QNAP这性价比高，普及率还非常好的NAS大概这时候就会走入视野。

那么问题来了，硬件性能好就一定好用么？
我司的一个十年前的群晖，因为CPU是x86，docker和套件找不到对应的(网上一堆喊着x86的实际是x86_64)，在其他人的帮助下，我最终在这台破群晖上成功的跑起来了QB的docker，虽然还有点小问题，但是QNAP呢？跑起来了，登不进去。

目前我用QNAP遇到的问题有

  1. 原系统DNS本机架了个dnsmasq，配置啥的都是锁死的，重启自动还原，然后不定时就对自己进行DNS投毒，给一些域名解析返回10.0.0.1。扩展阅读: https://www.google.com/search?q=dnsmasq+返回+10.0.0.1 （用了大概一年，QNAP才更新系统修正这个问题）
  2. 想改的话，web里配置没用，只能进ssh改(但是重启会还原，所以每次都要改)，但是改了之后就天天告诉你硬盘分析上传失败(虽然我也不在乎这个)
  3. qb、tr有了(de要钱，rt还是个pro版本)
  4. docker折腾了，进不去，各种界面不如群晖那么方便。
  5. 最后逼得我上了虚拟机，结果自带的vnc因为安全相关，不允许访问剪贴板。自带支持的spice也是同样。
  6. Nomachine相关问题: https://ted423.github.io/Software/Remote/NoMachine，目测QNAP虚拟机的VNC画面可能创建的比较特殊
  7. 很多东西都是重启会还原
  8. nfs server再检查完硬盘后需要ssh进去重启服务，不然会挂载不上
    简要的问题在于，你买了QNAP，系统就只能用QNAP的，当你发现你不喜欢这个系统的时候。就很尴尬了。
    在我的使用环境中，经常要ssh进去，比如结束进程，或者mount。但是QNAP又是有自己一套体系的，你没法确保他的命令和别的完全是一致的，效果也是一致的。比如docker，无形中增加了很多学习难度。
  9. 迁移共享文件夹，只给你ln过去，东西还在原来的盘上。什么傻逼系统

东西不是有就行了的，有些甚至你不知道他到底实现到什么程度。

目前大部分软件都是第三方开发的Qoolbox(qb升级换位置，然后变成了两个软件)，他要是哪天不干了就很麻烦。相对群晖那边会自己编译弄套件的似乎多些。

当然，QNAP 4盘位的空间利用率还是不错的，你弄个DIY的，电源就占很大位置。
可能还有别的解决方案，不过我只负责吐槽

QNAP 的 nfsv4 有问题，具体见 https://forum.qnap.com/viewtopic.php?f=35&t=165385#p813893

不幸的在于，autofs似乎完全没办法，只能用nfsv3 –2024-07-15

网络同步盘缺陷记录

优秀

  • dropbox
    同步不够实时(服务器端的改变并不会立刻同步回本地)
    如果你希望他立刻进行一次完全的同步，你需要退出他的客户端再重新打开。
  • Onedrive
    不支持局域网内同步,同步逻辑有缺陷，偶尔会干删一堆文件到回收站再重新下载的事(由于下调存储空间一事，已经不准备继续使用，准备转yandisk看看，15元/月才50G，我就呵呵了)
  • 快盘
    国内做的最好的同步盘，不过服务器实在…
    客户端弹窗简直烦人
    因为有秒传机制，会出现上传的某些文件永远无法下载（你也没法重传，因为秒传了）
    因为上一条，是否使用自己斟酌

可用

  • mega
    支持创建同名文件(夹)，不利于同步，同步依据文件修改时间，在修改时间不变改变文件的情况下会去下载源文件覆盖修改后的文件。
    在其他电脑上直接在网页端上传文件同步回本地后会再次进行上传（可能跟装了浏览器扩展有关），然后创建出同名文件夹(我这里甚至出现了重启次机子就会重新上传的状况)
  • Yandex
    同步时会导致同步文件夹内的文件写入慢，影响其他读取该文件夹的软件运行
    不支持回复文件历史
    * 新浪微盘同步盘
    同步时暴力拒绝其他修改，你可以把他理解为一个手动同步工具…
    需要修改时你就暂停他的同步吧。
    * 酷盘(已死)

不推荐使用

  • Google Drive
    这同步简直败坏谷歌的名声（虽说谷歌做的所有软件一直不是我喜欢的，因为感觉太偏执了）
    支持创建同名文件(夹)，不会进行匹配，所以多端同步做的简直shit,你只能让他自己去同步。如果你自己往里放文件协助同步你会发现服务端建立了一堆同名文件或文件夹，然后你会在另一端看到一堆wrong,逗我。
  • 360同步
    只支持单任务，在上传时居然会不去判断网盘内的修改，然后就出现了完全重复的两个文件。基本只是个玩具，根本算不上同步盘。如果你希望他把你的资料弄得乱七八糟，你可以使用他。
  • 百度云同步
    服务器相当不稳定，一般的同步问题都是服务器导致的，甚至有时明明同步未完成，网盘也会判断同步已完成。百度对部分视频进行了破坏性修改，突破了网盘的底线，因此不建议使用这个网盘存储自己的资料
  • 微云
    我当时用的时候，微云是文件名同步，即便改变文件内容，微云也不会去重新同步

不使用aria2c的原因

这已经是我第3次试图使用aria2c来代替IDM了，但最终我还是失败了。
想想还是记录下吧，避免以后再次忘了当初为什么放弃。
最早aria2c配置，自启动服务，扩展什么都折腾过。

原生的问题

  1. 下载时需要提供文件名，由于程序本身不具备GUI，又不支持自己生成文件名，对于很多要连接会跳转后通过特定post来提供文件名(例如百度网盘)。下载工具在使用上就会有很多限制，比较好的解决方式就是在浏览器上通过特定的工具来发送下载时的文件名，诸如chrome+aria2c integration之类。
  2. 但这些aria2c的补充通常不会有太强的完善性，因为那样无异于完全开发出一个下载工具，所以大部分GUI或者扩展之流也都会有些缺陷，诸如在路径的选择上。
  3. 多线程下载软件在不被服务器针对的情况下，其实很少在下载速度上有巨大的差别，可以说aria2c也并没有超出其他下载工具太多
  4. 这点才是我放弃的真正原因，就是aria2c取消任务后不会删除下载到一半的文件。

其实在我看来aria2c之所以流行是因为chrome的原因。因为chrome自身下载实在是很特别，chrome上实现接管下载的扩展往往不是那么好用并且稳定。

#Cent Broswer+aria2c integration

  1. 有时候扩展漏过了下载，没能传递到aria2c,浏览器自身开始了下载。
  2. 然后就是打开某些文件没法直接在浏览器看，会被下载接管
  3. 会干扰一些其他扩展的功能。

#Motrix
优点：开源，前面也说了，”但这些aria2c的补充通常不会有太强的完善性，因为那样无异于完全开发出一个下载工具”

  1. https://github.com/agalwood/Motrix/issues?q=is%3Aissue+author%3Ated423+is%3Aclosed PAC那个是我提错了，但别的属于没法满足我的使用习惯的。
  2. https://github.com/agalwood/Motrix/issues/created_by/ted423 这是还开着的。
  3. 抛开BT相关，各种功能并没法完全打败IDM。(例如IDM的队列可以设置下载间隔，遇上那种防批量下载的服务器就会很有用)

干扰其他功能和浏览器原生功能我是不能忍得，最主要这是一种替代的不完美实现，何必呢。
不如直接点。
另一方面 cent broswer自身的内置下载在选取询问路径的情况下是可以调用迅雷或者IDM的

像我装了QB，IDM，迅雷，Motrix。
迅雷和QB都是最常开，剩下两者IDM作为补充，Motrix(aria2c)就很少开了
