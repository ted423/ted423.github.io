不建议使用Refs

نُشر في 2021-07-15 عُدل في 2021-12-22 في Software

https://docs.microsoft.com/zh-cn/windows-server/storage/refs/refs-overview

目前 REFS 上未提供以下功能

功能 ReFS NTFS
文件系统压缩 否 是
文件系统加密 否 是
事务 否 是
对象 ID 否 是
卸载数据传输 (ODX) 否 是
短名称 否 是
扩展的属性 否 是
磁盘配额 否 是
可引导 否 是
页面文件支持 否 是
在可移动媒体上受支持 否 是

以下功能仅可用于 REFS

功能 ReFS NTFS
块克隆 是 否
稀疏 VDL 是 否
镜像加速奇偶校验 是（在存储空间直通上） 否
文件级快照 是1 否

其他不说就算了，引导啥的可有可无，毕竟Refs目标是存储用的文件系统。
结果你不支持压缩….

举个例子吧，zfs 上 10t 压缩率 1.44x(当然要看文件)，原本是14T。
那么问题来了，NTFS也支持Dedupe
甚至磁盘配额是AD共享文件服务器非常重要的功能。

不过需注意的是开了整盘压缩后写入速度会降非常多，大概只有30MB/s。所以建议不要全开，仅对可压缩的文本之类的开下就好了。或者干脆7z会更好

想必是电脑快爆炸了。这可把我吓得不行。
然后呢，我的注意力就转移到了这“安全和维护”上。这破玩意到底在哪呢？
我宇宙无敌人间第一人性化操作系统Windows才不会告诉你我把他放哪了，少年啊，去寻找吧。

我找了半天终于在计算机右键属性里找到了这玩意(其实找了两次，第一次找到后气的不行然后关了。结果忘了这东西在哪又找了一次。)
然后发现一点都不”关键”，因为我关了UAC。
电脑是否需要启用UAC我在这里不再做讨论。毕竟Win10系列是你不开UAC通知，但UAC仍然有部分在那里。以至于管理员权限运行的软件读不到网络硬盘。

总而言之我就是折腾了半天发现这是个屁大的小事。（可以设置取消安全和维护对UAC的检测）

[不再推荐]Slax - Linux PE

نُشر في 2021-02-24 عُدل في 2021-12-22 في System

不推荐的理由在于Ubuntu和Manjaro的安装盘就能做到，并且一般效果比这个还要好
2年未维护，似乎没有提供源码

其实要有个WinPE能看Linux的文件系统也就不用这玩意了
https://www.slax.org

但目前来讲这东西还算有用。
修复了一个ESXi的文件损坏的问题

请注意这个iso不是像别的一样拿什么东西写到移动硬盘或是U盘上，而是直接解压（NTFS文件系统都行）,移动硬盘或U盘中可以保留slax这个文件夹（直接放进去），然后跑boot里的bat。

另外注意这个对移动硬盘可能相性不好，也可能是文件系统的锅，建议优先U盘

请勿购买QNAP

نُشر في 2021-07-03 عُدل في 2021-11-24

本文属于用啥黑啥系列。
在专用NAS中，如果想用那些某类玩家都用的软件，其实可选择性不多。
像什么西数，铁威马，别人跟你说某软件可能用不了，估计你的心就凉了大半。
然后第二遇到的大概就是群晖的贵了。

因此留给你的选择大概不多，QNAP这性价比高，普及率还非常好的NAS大概这时候就会走入视野。

那么问题来了，硬件性能好就一定好用么？
我司的一个十年前的群晖，因为CPU是x86，docker和套件找不到对应的(网上一堆喊着x86的实际是x86_64)，在其他人的帮助下，我最终在这台破群晖上成功的跑起来了QB的docker，虽然还有点小问题，但是QNAP呢？跑起来了，登不进去。

目前我用QNAP遇到的问题有

  1. 原系统DNS本机架了个dnsmasq，配置啥的都是锁死的，重启自动还原，然后不定时就对自己进行DNS投毒，给一些域名解析返回10.0.0.1。扩展阅读: https://www.google.com/search?q=dnsmasq+返回+10.0.0.1
  2. 想改的话，web里配置没用，只能进ssh改(但是重启会还原，所以每次都要改)，但是改了之后就天天告诉你硬盘分析上传失败(虽然我也不在乎这个)
  3. qb、tr有了(de要钱，rt还是个pro版本)
  4. docker折腾了，进不去，各种界面不如群晖那么方便。
  5. 最后逼得我上了虚拟机，结果自带的vnc因为安全相关，不允许访问剪贴板。自带支持的spice也是同样。
  6. Nomachine相关问题: https://ted423.github.io/2020/12/11/Software/NoMachine/，目测QNAP虚拟机的VNC画面可能创建的比较特殊
  7. 很多东西都是重启会还原
  8. nfs server再检查完硬盘后需要ssh进去重启服务，不然会挂载不上
    简要的问题在于，你买了QNAP，系统就只能用QNAP的，当你发现你不喜欢这个系统的时候。就很尴尬了。
    在我的使用环境中，经常要ssh进去，比如结束进程，或者mount。但是QNAP又是有自己一套体系的，你没法确保他的命令和别的完全是一致的，效果也是一致的。比如docker，无形中增加了很多学习难度。

东西不是有就行了的，有些甚至你不知道他到底实现到什么程度。

目前大部分软件都是第三方开发的Qoolbox(qb升级换位置，然后变成了两个软件)，他要是哪天不干了就很麻烦。相对群晖那边会自己编译弄套件的似乎多些。

当然，QNAP 4盘位的空间利用率还是不错的，你弄个DIY的，电源就占很大位置。
可能还有别的解决方案，不过我只负责吐槽

[问题处理]SVN踩得几个坑

نُشر في 2021-07-29 عُدل في 2021-11-22

The server sent a truncated HTTP response body

https://www.jianshu.com/p/ad53304dbdbf
印象里这篇文章似乎没啥用，后面是要改apache的设置。

备份

大部分文章都强调不能cp和rsync。但结果来说，rsync可能会成为最终结果

  1. SVN并没有提供直接备份所有repo的办法，所有文章的repo都是基于单个repo的，所以要备份只能遍历。
  2. svnadmin dump 出来的文件比服务器上的repo还要大。
  3. svnadmin dump 的某个repo只有1kb，空repo也是1kb。手动备份能正常备份，所以记得一定要开log
  4. svnadmin hotcopy卡住，网上看跟文件系统有关，大概我用的NTFS(NTFS ,samba都不行，只能nfs，samba不行的原因在于某文件创建后会卡住无法删除，改用nfs挂载同目录后正常)

验证相关

1.8 新增了一个设置，默认关了权限继承
应该说最新的跟1.6不一样，[/]给的权限无法继承，原因不明

恢复相关

硬盘炸了，原来项目备份时报错”serialized hash missing terminator”
导致项目备份数据不对，恢复后无法使用。期间报错”打开文件系统“db”的BDB环境句柄时出错”(Berkeley DB error for filesystem DB_CONFIG)，”svn E160004 current file”(这个是恢复不了我瞎改导致的，实际文件正常的话recover会自动修正current,手动修改没意义，对不上就会报错)。”svn E200002 malformed representation for node-revision”
经确认是有报错之后，备份时丢失了format以及db文件夹下的一些数据
正常项目的db文件夹下，revs和revprops都会多套个叫0的文件夹(0-999放到0，1000-1999放到1这样)
revs里最大的数字为最终版本号。(不过这个不重要)
主要db文件夹下会需要fomart为(我这里是)

而报错导致的备份的fomart都是2
还需要一个”min-unpacked-rev”(这个从正常项目拷一个塞进去就行)
然后svnadmin recover和svnadmin verify就行，但如果期间版本有问题，那就是另外一个问题了
