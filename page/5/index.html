下载、资源类软件缺陷记录

نُشر في 2020-06-19 عُدل في 2021-04-24 في Software

BUG

  • aria2c https://ted423.bitcron.com/post/software/zi-yuan-lei/aria2c
  • Transmission
    1. tracker连接性不强，获取tracker信息过慢
    2. 制作种子不支持网络映射
    3. 单线程，牺牲性能降低占用(理论我觉得rt会更好，不过rt不支持改名文件夹这些，并且两个长期不更新的软件，tr反倒是最后更新了一下)
    4. 部分种子识别有问题，部分种子无法使用(我提issue被ban)
  • qbittorrent
  1. 制作的种子dmhy无法读取中文文件名(但应该是网站问题)
  2. 制作的种子tracker中必须隔开一行，否则traker其他软件无法识别
  3. Others https://github.com/qbittorrent/qBittorrent/issues/created_by/ted423
  • 迅雷
  1. 做种相当week,特别是点击添加重新下载时会直接把已下载文件从磁盘抹掉
  2. 现在的版本不能从别的地方下好然后拉进去补充进度了
  3. 迅雷并不会显示具体的tracker信息，并且也不会具体的显示谁给你传数据，也不会显示你在给谁传数据，个人感觉它是努力淡化BT原理，营造一种速度都是迅雷提供的错觉，所以很多迅雷用户体现一种反智现象
  • Deluge
  1. 对中文支持不好,添加种子时路径有中文就出错了，但BT下载的文件夹又可以是中文
  2. 2.0.3大概是不准备也没法支持Windows了

【迅雷11】是继【迅雷X】之后的换代版PC迅雷客户端。 在【迅雷11】上我们重新设计了主界面框架，许多忠实雷友们过去一直吐槽的太像浏览器的问题，现在已经得到解决。

【迅雷11】的最大亮点，是我们开创性的将【下载】与【云盘】合二为一。

当您下载网络中的文件时，可以灵活的选择将文件下载到【电脑】或者【云盘】。

下载列表

首先要从下载列表说起，在这个版本中，我们将下载列表的任务分类，由原来【迅雷9、迅雷X】采用的纵向排列，改为横向排列。这是由于纵向排列时，每次需要切换任务分类，都要在上下分类之间长距离的移动鼠标，使用较为不便。

悬浮窗

悬浮窗也迎来了变化，从【迅雷7】一直用到【迅雷X】的圆形悬浮窗，现在终于改版啦！

迅雷11新的悬浮窗看上去像是个立方体，当有正在下载的任务时，会在右侧展开一个简短的进度条。

这个全新的设计可以更清晰的显示下载进度，视觉效果也炫酷不少哦。

立即添加（到云盘）

如果您是会员用户，当您选中任意一个下载中的任务时，在下载任务的下方会出现一个新的扩展栏，可以让您快捷的将任务添加到云盘里。

下载列表展示云盘取回

在迅雷11过去的几个版本中，从云盘取回的任务，都只展示在云盘模块的【传输列表】里，对于经常需要查看云盘取回文件的用户来说颇为不便。现在通过下载列表也可以查看云盘取回的文件了。

#更新信息

  • 下载列表的任务分类由纵向排列改为横向排列
  • 采用全新视觉设计的悬浮窗
  • 新增【手机扫码免费加速】功能
  • 左侧栏【下载、云盘】右侧增加数字红点（可展示当前正在下载的任务数，以及正在云添加和上传的文件数）
  • 下载列表中视频任务下方增加【云盘播放】功能
  • 下载列表中非视频任务下方增加【立即添加（到云盘）】功能
  • 下载列表中支持展示从云盘取回的文件
  • 新增【云盘视频智能缓存】功能，改善云盘视频播放体验（可在设置中心停用）
  • 修正某些情况下，播放组件下载解码器失败的问题
  • 云盘增加文件【举报、申诉】功能
  • 云盘传输列表入口移至顶部
  • 云盘回收站改为独立对话框

落尘之木优化内容

  1. 本地VIP专享悬浮窗
  2. 移除发现栏
  3. 精简多余菜单
  4. 自动识别VIP用户，非VIP直升本地超级会员
  5. 移除下载区文字广告
  6. 移除头像悬浮广告
  7. 禁止更新版本

[Git]一些命令

نُشر في 2020-04-21 في Code

如无特别说明，本人的所有Git操作都是以Github为主

git 文档
https://git-scm.com/docs

本地分支删除

查看项目的分支们(包括本地和远程)

删除本地分支

reset 删除错误提交的commit

使用这个的前提条件是没有push,一般不建议push -f
git reset --hard <commit_id> 回退到某一版本。
git reset --hard head~1 回退一个版本(数字2自然就是两个版本啦)
然后 git push --force

rebase 伪造历史

git rebase -i HEAD~10 (修改最近10个历史)

下面是rebase打开的文本里命令的提示(git是通过判断进程创建以及进程是否仍然存在来判断用户是否开始、结束编辑，所以多标签的文本编辑器会影响git判断你是否还在编辑)

其中每个历史里的变动占一行，我们只要改前面的pick即可，其中主用的一般是squash，squash会合并改动，再次rebase时是不会拆分的。edit则是修订历史变动里的错误。也可以改动这些变动的顺序。

需要注意的是git里以记录改动为主，如果修改历史变动是会改变现有代码的(特别是你把某个文件的最后一次commit给弄没的时候)。drop 意味着当时进行的修改也会被取消。很容易造成冲突(conflict)，
所以一般是建议squash来合并commit

最后 git push --force
这个命令会把远程的彻底抹掉，出问题也会很彻底的，所以要谨慎。
