[问题处理]SVN踩得几个坑

发表于 2021-11-22

The server sent a truncated HTTP response body

https://www.jianshu.com/p/ad53304dbdbf
印象里这篇文章似乎没啥用，后面是要改apache的设置。

备份

大部分文章都强调不能cp和rsync。但结果来说，rsync可能会成为最终结果

  1. SVN并没有提供直接备份所有repo的办法，所有文章的repo都是基于单个repo的，所以要备份只能遍历。
  2. svnadmin dump 出来的文件比服务器上的repo还要大。
  3. svnadmin dump 的某个repo只有1kb，空repo也是1kb。手动备份能正常备份，所以记得一定要开log
  4. svnadmin hotcopy卡住，网上看跟文件系统有关，大概我用的NTFS(NTFS ,samba都不行，只能nfs，samba不行的原因在于某文件创建后会卡住无法删除，改用nfs挂载同目录后正常)

验证相关

1.8 新增了一个设置，默认关了权限继承
应该说最新的跟1.6不一样，[/]给的权限无法继承，原因不明

恢复相关

硬盘炸了，原来项目备份时报错”serialized hash missing terminator”
导致项目备份数据不对，恢复后无法使用。期间报错”打开文件系统“db”的BDB环境句柄时出错”(Berkeley DB error for filesystem DB_CONFIG)，”svn E160004 current file”(这个是恢复不了我瞎改导致的，实际文件正常的话recover会自动修正current,手动修改没意义，对不上就会报错)。”svn E200002 malformed representation for node-revision”
经确认是有报错之后，备份时丢失了format以及db文件夹下的一些数据
正常项目的db文件夹下，revs和revprops都会多套个叫0的文件夹(0-999放到0，1000-1999放到1这样)
revs里最大的数字为最终版本号。(不过这个不重要)
主要db文件夹下会需要fomart为(我这里是)

而报错导致的备份的fomart都是2
还需要一个”min-unpacked-rev”(这个从正常项目拷一个塞进去就行)
然后svnadmin recover和svnadmin verify就行，但如果期间版本有问题，那就是另外一个问题了

压制组相关

发表于 2019-06-25 更新于 2021-10-09

现在跑去压制，所以稍微多了点了解
在压制正常处理的情况下（同一编码器），码率基本决定画质，剩下的区别在于画面的瑕疵修复，这种属于主观观感。
一部影片所需的码率跟画面大小是正相关的。如果你要挑个3000kbps视频码率的电影放35寸电脑显示器(或者70寸电视)上看然后嫌别人画面差的话，那主要应该是你的脑子的问题。

一般来说压制也有分流派
例如iNT组主要都是尽量靠近原生影片，但是码率最高要求是源片的75%以内就好。他们一般不降噪，所以一部片如果噪点太多的话，可能只会有720p的压制。通常会使用兼容性最好的音轨。最新的标准是DDP(EAC3)5.1 1024kbps,但是以前的要么是DTS 1509kbps(略强于AC3 640kbps)要么是DD(也就是AC3)。

二次元组多为尽量低的码率并尽量修复瑕疵。并且由于日韩中经常有奇葩原盘，所以大部分二次元压制组有顶尖的瑕疵修复技术，但他们的作品不被各个三次元国外高清站所接受。基本是不符合站点规范。音轨要么AAC(压缩率高的有损音轨，320kbps,但音质比肩DTS 1509kbps),要么flac5.1(各种不兼容，PS:flac 无损压缩率最高，通常DTS最差，turehd还行，但是兼容性差，压制放tureHD还必须带条AC3 5.1的兼容音轨)
当然啦，以BD来说，一季动画200G，收齐可能半个T，12T硬盘你大概收不了多少东西。甚至你眼睛被原盘弄瞎都有可能。

当然还有随心所欲组和乱压组。有几个公网知名的，边比别人多切几条(脏边不修直接切，然后被喷技术不行)，分辨率押错了的。

另外三次元原盘一般比二次元问题要少,基本只要deband就行。当然我猜二次元组遇到这种可能不会像金种标准那样一帧一帧修。

睡眠东西保存在内存中，一旦彻底断电电脑状态会丢失，硬盘因为彻底关了，断电应该不会伤硬盘(以前有很多笔记本合上盖子进睡眠之后，再开机可能因为硬件唤醒之类的会起不来，当时似乎都是更驱动或者换主板解决的)
休眠是把内存状态那些塞到硬盘里，如果你是固态的话，就很快，但机械的话，可能会要很久，无论进入休眠和退出休眠都是。好处是彻底的断电，理论和关机一样。不过在大内存的机子就比较惨了，因为是完全保存内存的状态，再加上虚拟内存一般会设的跟内存差不多大。(32G内存机子系统盘没空间也是挺2333的)。
另外还有个混合式睡眠，似乎需要开休眠才看得到。不过感觉Win10可能把这个干掉了？另外还有Fast Startup的东西。
有需求的自己查下吧，我个人猜是先进睡眠，然后把内存的东西再写到硬盘上。
超大内存(200+G)的服务器装Win10会出问题，休眠总会自己打开，然后把C盘用光，虽然hiberfil.sys只有90G。但我也不确定这是不是偶然的
补充一点，Sever默认是没休眠的。甚至默认虚拟内存都小
