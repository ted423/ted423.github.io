也许要再见了，Bitcron&Farbox

প্রকাশিত হয়েছে 2021-05-02 সম্পাদিত হয়েছে 2021-08-05

今天检查博客没同步到最新状态才看到Farbox 2.0。
变动很大，思虑良久，我开始考虑彻底的离开。

我使用Farbox最早是2014年，毕业没多久。被Farbox的展现出来的世界所触动。
这几年，分享、笔记、边写边思考，算我在Bitcron&Farbox的主要工作。
曾试过Wordpress，不过白嫖的服务器几下就炸的不成样子。
选择Farbox的原因之一便是我真的很喜欢Dropbox ->https://ted423.bitcron.com/post/software/zi-yuan-lei/tong-bu-pan-ping-jie
所以无论是Farbox的”『简单』才持久”，”使用者自己拥有数据”，都让我非常认同。(Dropbox后端的版本控制，曾让我解决了单机游戏误操作的问题，也解决过自己弄乱的环境)。

看Farbox的博客也是我的一大爱好，在我的认知中，Farbox团队一直是一个有理念、有思维的团队。
我一直很反感完全的程序员思维，这种思维在开源软件里展现的非常多。
所以”为什么用户不知道自己需要什么”，”写代码的产品经理们”，”设计的容量”让我非常认同。

但现在Dropbox没了，很奇妙的一件事，却让我感到一切都不一样了。

简单么？ -> 或许对于Mac用户来说还是很简单(我有虚拟机黑苹果，也有能拿来玩的Macbook)，但我觉得现在不那么简单了。我一直都知道，同步逻辑是比较困难的，所以憨憨谷歌的同步盘以及书签同步跟屎一样。版本控制也是如此，git或dropbox在这块算是比较好的解决方案。
另一方面，手动同步跟静态更新又有什么区别呢？
使用者自己拥有数据 -> 自己手里拥有数据跟定时备份区别很大么？在我看来区别不大。
当然现在开源了，或者这个项目没思考这个转变意味着什么。这个特性目前已经不太适用。

在Farbox 2.0不再有Dropbox之后
我大概只能手动运行py脚本(甚至可能要自己搭py环境，然后换台电脑估计还得重新弄，相信我，我折腾过VapourSynth的环境，不麻烦但也绝对不算简单)，或者折腾个定时任务跑同步。以我的用法，大概手动是最合算的，但是就很怪。就如同前面说的。”自己手里拥有数据跟定时备份区别很大么？在我看来区别不大”，官方做个客户端来做同步对用户应该是更好以及更简单的。

也许，Farbox 2.0的网页端后台足够好用？但目前并没有一个足够分量的介绍和试用。

或者自己搭个服务器，折腾个自动同步或自动备份。

或者我仍能使用Dropbox来解决数据和版本控制的问题，一个python对于闲着无聊写vpy的我来说，也并不算太大的麻烦。但扪心自问，这样优雅么？
我放弃了aria2c+浏览器扩展是如此，放弃了tera term是如此。我现在不喜欢找替代的解决方案，因为这不优雅，也不是我所追求的。

我感觉，”『简单』才持久” 这个理念已经被Farbox抛弃了，当然，或许所谓的开源，其实是放弃了整个项目。或许这才是正解，所以留下了让用户手动py同步这种使用方式。开源以后，数据存放在哪里也不再重要，由用户自己决定，反过来说，开源以后的使用模式跟之前保持一致其实是不太合适的。

一些早就不满的点

  • 在我使用中，复杂的Markdown解析经常出问题。甚至有时第一时间看显示正常，过一段时间就全乱了。甚至同样写法，有的可以，有的就出问题，反观目前的Hexo，有问题就一直有，有问题也能分析出原因出来。
  • 博客一直不显示修改后的结果也让人不耐。
  • 文档乱放，https://pi.bitcron.com/ 以及 https://api.bitcron.com/read/h 入口位置就很奇怪，模板一开始说开源，然后最新的却不公开源码，想实现一些类似功能无从参考，api什么的不明晰，也无从调试。
  • 2021-06-17 才发现本地文件丢失，但是网页还正常显示在，说明Bitcron的Api一顿瞎操作把Dropbox上的文件写没了(甚至更早期的Farbox里的相同文章都还在)。毕竟人家说了 “Dropbox 的关联和后续可能失效，Bitcron 在 2021 年的时候就已经基本冻结代码了（简而言之就是我们什么都没动 Dropbox 的 API 就失效了），之前一次 Dropbox API 的巨大变化，导致当时的 FarBox 1.0 匆忙下上线了 Bitcron，没有再为 Bitcron 对应 Dropbox 新的 API 的计划。”. 印象里这几篇文章用Bitcron的在线编辑器写过(“使用者自己拥有数据”,balabalabala)。总之自认倒霉吧，反正应该还能找回来。
  • 2021-08-05 Site Suspended了，我想到上面的情况，然后我还在咕着没把文章搬完就菊花一紧，也还好后台的文件管理器数据还在，然后我又一个误操作差点干掉一篇文章，回头一看farbox文件夹里的那篇文章还在。吓出一身冷汗。

我习惯从根源上解决问题，所以，大概再见了。
寻找下一个符合我喜好的博客或干脆彻底的解放自己。

bit Byte bibyte(雾) 存储单位，速度单位的一些说明

প্রকাশিত হয়েছে 2022-03-27 শ্রেনী Document

在计算机中 b=bit B=Byte
其中 1Byte=8bit
B=Byte属于习惯用法，并不表示byte在使用中必须大写。
但是在计算机使用中，KB, MB，GB，大小写必须写对，否则意味这个人技术不过关，基础太烂。
另，MB/s=MBps=8Mbps，通常说的百兆带宽是100Mbps也就是12.5MB/s。
当然百兆带宽属于口语化用词，你不用纠结一个人说千兆带宽下载有100兆（100MB/s）这件事。
当然360浏览器下载速度是M/s的事你也不用纠结，因为说明360浏览器的下载器开发者水平太烂。

这时候冒出来个MiB(Mebibyte)就需要仔细讲讲了
Multiple-byte units
Mebibyte 记作 MiB ，由国际电工委员会（IEC）于1998年制定。这个单位被设计用来某些时候替代MB（megabyte），因为在计算机相关内容中MB有可能被用来等于2的20次方，虽然数值很相近，但MiB与国际单位制（SI）中的MB（10的6次方）还是有严格的区别。
这个单位不常用，不过你知道10TB在Win上只有8.92TB(实际是TiB)就是这个单位做怪了。当然，离不开本博客口头语，微软傻逼。（所以说为啥微软某些地方这么像国内软件啊，不过确实比360强点）

多网络环境可以进行路由优先级调整，避免类似不能访问外网的路由优先级高导致电脑不能上网
其实调整跃点就可以
对于内部的，不跨网段的，其实可以把网关删掉

希望优先级低的，在适配器处，进入ipv4设置，然后点击高级，取消自动跃点，给跃点数填个很高的值

route print
可以看到路由优先级

可以手动指定路由
route add 192.168.0.0 mask 255.255.255.0 192.168.68.254

-p 代表是永久路由

X.509

https://zh.wikipedia.org/wiki/X.509

  • CA(Certificate Authority)被称为证书授权中心，是数字证书发放和管理的机构。
  • 根证书(root certificate)是CA认证中心给自己颁发的证书,是信任链的起始点。安装根证书意味着对这个CA认证中心的信任。
  • 数字证书颁发过程一般为：
    1. 用户首先产生自己的密钥对，
    2. 将公共密钥（公钥）及部分个人身份信息传送给认证中心
    3. 认证中心在核实身份后，将执行一些必要的步骤，以确信请求确实由用户发送而来，然后，认证中心将发给用户一个数字证书，该证书内包含用户的个人信息和他的公钥信息，同时还附有认证中心的签名信息

证书分为 OV DV EV
EV SSL: 扩展验证型SSL(Extended Validation SSL)、OV SSL: 机构验证型SSL(Organization Validation SSL)、DV SSL: 域名验证型SSL(Domain Validation SSL)

DV不用做验证，比较好签发，不过也容易被仿冒。
个人用，不是很重要的用途可以使用。

OV 是公司组织验证，像GoDaddy会要你的增值税发票，然后打你公司电话

EV 是更近一步的
支持多子域名

格式说明
PKCS 全称是 Public-Key Cryptography Standards ，是由 RSA 实验室与其它安全系统开发商为促进公钥密码的发展而制订的一系列标准，PKCS 目前共发布过 15 个标准。 常用的有：
PKCS#7 Cryptographic Message Syntax Standard
PKCS#10 Certification Request Standard
PKCS#12 Personal Information Exchange Syntax Standard
X.509是常见通用的证书格式。所有的证书都符合为Public Key Infrastructure (PKI) 制定的 ITU-T X509 国际标准。
PKCS#7 常用的后缀是： .P7B .P7C .SPC
PKCS#12 常用的后缀有： .P12 .PFX
X.509 DER 编码(ASCII)的后缀是： .DER .CER .CRT
X.509 PAM 编码(Base64)的后缀是： .PEM .CER .CRT
.cer/.crt是用于存放证书，它是2进制形式存放的，不含私钥。
.pem跟crt/cer的区别是它以Ascii来表示。
pfx/p12用于存放个人证书/私钥，他通常包含保护密码，2进制方式
p10是证书请求
p7r是CA对证书请求的回复，只用于导入
p7b以树状展示证书链(certificate chain)，同时也支持单个证书，不含私钥。
